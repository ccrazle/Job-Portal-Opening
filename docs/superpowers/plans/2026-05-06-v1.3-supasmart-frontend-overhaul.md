# v1.3 — Supasmart Frontend Overhaul: Safe Deployment Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Replace the vanilla-JS frontend with the Supasmart React/Babel frontend (same HTML-only format), fully wired to the existing Express/Supabase backend — without touching any existing database data, user-created job openings, or screening/outreach history.

**Architecture:** The frontend file (`frontend/index.html`) is the only file being fully replaced. The backend (`server.js`, `db.js`) remains 100% unchanged. All user data lives in Supabase's `key_value_store` and `resume_files` tables and is never written to or migrated by this change — the new frontend simply reads/writes the same API endpoints the old one did.

**Tech Stack:** React 18.3.1 (CDN), Babel Standalone (in-browser JSX), Tailwind CSS (CDN), Express backend (unchanged), Supabase PostgreSQL (unchanged)

## Safety Guarantees — Critical

| Risk | Why it's safe |
|---|---|
| User's live job openings (OG-2026-003/004/005 etc.) | Stored in Supabase `key_value_store` table under keys `jp_posted`, `jp_drafts`. No backend change. New frontend reads the SAME keys via the SAME `/api/store/:key` endpoint. |
| Screening history & candidate scores | Stored under `jp_screening_{jobId}` keys. No schema change. New frontend reads them the same way. |
| Outreach history | Stored under `jp_outreach` key. Same pattern. |
| Resume PDFs (Supabase `resume_files` table) | Table untouched. New frontend still calls `GET /api/resumes/:jobId/:candidateId`. |
| localStorage caches (`jp_pdfs_*`, `jp_token`) | Browser-side. Unaffected by a frontend file swap. Auth token persists. |
| Backend server.js | **NOT MODIFIED** — zero changes. All API routes stay identical. |
| Backend db.js | **NOT MODIFIED** — zero changes. |
| Database schema | **NOT MODIFIED** — no migrations, no CREATE TABLE, no ALTER TABLE. |
| `.env` / credentials | **NOT COMMITTED** — `.env` is in `.gitignore`. |
| `Skill Prompts/leave-credit.md` | **NOT COMMITTED** — excluded from staging. |

## Execution Strategy

### Phase 1: Prepare the New Frontend File

The new `frontend/index.html` will be the Supasmart-full.html file with these additions:

1. **API Integration Layer** — A `<script>` block (before React CDNs) containing:
   - `window.AUTH` object for token management
   - `window.db_api` object with `get()`, `set()`, `remove()` methods → calls `/api/store/:key`
   - `window.resumeApi` helper for PDF upload/retrieval
   - `window.skillApi` helper for AI scoring via `/api/run-skill`

2. **SignIn Component** — Wired to `POST /api/auth/login` (real auth, not mock)

3. **App Component** — Checks `GET /api/auth/me` on mount, redirects to sign-in if 401

4. **Hire Component** — Loads openings from `db_api.get('jp_posted')` and `db_api.get('jp_drafts')` on mount

5. **HireScreening** — Calls `POST /api/resumes/bulk` for upload, `POST /api/run-skill` for scoring

6. **HireOutreach** — New tiered system (from Supasmart-complete), persists to `db_api.set('jp_outreach', ...)`

### Phase 2: Commit & Push

```
git add frontend/index.html CHANGELOG.md
git commit -m "feat: v1.3.0 — Supasmart frontend overhaul with tiered outreach"
git tag v1.3.0
git push origin main --tags
```

### Phase 3: Hostinger Deployment

Since the server is on Hostinger and pulls from GitHub:
- Either the Hostinger deployment auto-pulls from `main` (if CI/CD is set up)
- Or a manual `git pull` on the Hostinger server updates `frontend/index.html`
- Backend process restart is **NOT required** (only frontend HTML changed, served as static file with no-cache headers)
