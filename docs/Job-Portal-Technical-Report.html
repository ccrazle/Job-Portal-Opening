Executive Summary

The Job Portal is an internal recruitment operations platform built for One Group's HR team. It consolidates the entire hiring workflow — job creation, multi-platform distribution, AI-driven resume screening, tiered candidate management, and outreach tracking — into a single system.

The product was designed to solve a specific operational bottleneck: HR was spending 3–4 hours per opening manually reviewing resumes that could be filtered in under 2 minutes by AI. The portal eliminates that gap.

The Problem We Set Out to Solve

Before this portal, the hiring process at One Group looked like this:

⚠

Manual Resume Screening

HR read every resume one by one, judging fit based on intuition. No standardised criteria. No consistency between reviewers. High-quality candidates were missed, and weak ones slipped through.

High Effort, Low Accuracy
⚠

Scattered Tooling

Job descriptions lived in Word documents. Candidate tracking happened in Excel. Outreach was managed over WhatsApp. Nothing was connected, and data was lost between handoffs.

Fragmented Workflow
⚠

Inconsistent Job Postings

Each time a role was posted, someone had to manually copy information from the JD document, reformat it for each platform, and hope nothing was mistyped. Different postings for the same role often had different details.

Error-Prone
⚠

No Institutional Memory

When a position was reopened months later, previous screening results, candidate notes, and outreach history were gone. Every hiring round started from zero.

Data Loss
Core insight: The bottleneck wasn't finding candidates — it was evaluating them. HR already had access to resumes. What they lacked was a fast, reliable, and consistent way to determine which ones were actually worth calling.

JD Engine — Verified Job Data

Most recruitment tools require you to type out every field from scratch each time you post a role. This portal takes a different approach: all job data is pre-loaded from your company's official JD documents, so creating an opening is a selection task, not a data-entry task.

Why This Matters

In an early version, the portal's database was populated with placeholder data that didn't match the actual JD documents. The "Billing Engineer" role, for example, showed Finance as the department and Mumbai as the location — when the actual JD document says Civil / Billing and The Saavira, Sector 48, Gurugram. Every auto-populated field was wrong.

This was a critical data-integrity issue. The entire value proposition of auto-fill depends on the underlying data being correct. We resolved this by extracting data directly from the official source documents.

AI Resume Scorer — The Core Capability

This is the product's primary value driver. Instead of HR spending hours reading through resumes, the portal uses AI to evaluate every candidate against the job description — applying the same strict, consistent criteria every time.

How It Works

1

Upload Resumes

Drag-and-drop up to 50 PDF resumes at once. The portal previews the file list before processing begins.

2

Server-Side Text Extraction

Each PDF is parsed on the server (not in the browser) for reliable, accurate text extraction. This solved early issues where browser-based parsing returned garbled data.

3

AI Evaluation (DeepSeek)

Resume text + job description are sent to the AI. It first parses the JD into weighted requirements, then applies two eligibility gates, then scores qualified candidates across four categories.

4

Ranked Results with Tier Assignment

Scored candidates appear in the screening table, automatically ranked and assigned to outreach tiers. No manual data entry required.

The Scoring Model

The AI follows a strict, multi-layered evaluation process that mirrors best-practice recruitment assessment. It was calibrated against manual screening results to ensure accuracy.

Layer 1 — JD Requirement Parsing

Before evaluating any resume, the AI reads the full job description and classifies every requirement into priority tiers:

Tier Signal Words in JD Weight
Must-Have must, required, mandatory, essential, minimum, necessary 3x
Should-Have should, preferred, expected, ideally, strongly preferred 2x
Nice-to-Have optional, nice to have, bonus, plus, desired, advantageous 1x

Requirements are then grouped into clusters — Technical Skills, Domain Knowledge, Experience, Education, and Soft Skills — to enable transparent, category-level assessment rather than a single opaque score.

Layer 2 — Eligibility Gates (Hard Pass/Fail)

Before any scoring begins, every candidate must pass two mandatory gates. These are binary — pass or fail. No partial credit, no exceptions.

✖

Gate A — Minimum Experience

Does the candidate have the minimum years of relevant experience stated in the JD? "Relevant" means the same domain — civil experience does not count for a mechanical role. Below the minimum = immediate rejection.

Binary Gate
✖

Gate B — Education Qualification

Does the candidate hold the exact degree level and field of study required? A Diploma does not satisfy a B.Tech requirement. The field must also match — B.Tech Civil does not satisfy B.Tech Mechanical. Unmet = immediate rejection.

Binary Gate
Why hard gates? Without them, the AI tends to give partial credit to unqualified candidates — a Diploma holder might score 60–70 for a B.Tech-required role because they have relevant skills. The gates ensure these candidates are rejected outright (score 0), matching what a careful human reviewer would do. In testing, this brought AI results in line with manual screening: out of 17 candidates for a QC Engineer role, both the AI and manual review identified the same 5 as qualified.

Layer 3 — Weighted Scoring (Qualified Candidates Only)

Only candidates who pass both gates proceed to scoring. The final score is built from four additive categories:

Category Max Points What It Measures
Skills Match /60 How well the candidate's skills overlap with the JD. Must-have skills (3x weight) matter far more than nice-to-haves (1x). Includes both keyword matching and contextual assessment.
Role Alignment /25 Has the candidate done similar work in a similar capacity? Checks actual responsibilities, not just matching job titles.
Experience Depth /10 Extra relevant experience beyond the minimum gate. 8 years when JD asks for 5 scores higher than exactly 5. Considers career progression and domain depth.
Education Strength /5 Higher qualifications, relevant certifications, or closely matched specialisations beyond the minimum requirement.
Scoring Formula
Skills (/60) + Role Alignment (/25) + Experience Depth (/10) + Education (/5) = Final Score (/100)

Score Interpretation

Range Classification What It Means
70 – 100 Strong Match Strong skills overlap, relevant experience, right education. Genuinely fits the role.
40 – 69 Partial Match Some requirements met but notable gaps. May be worth considering depending on the candidate pool.
0 – 39 Poor Match Minimal relevance to the JD. Passed the gates but skills/experience don't align well.
0 (Rejected) Did Not Qualify Failed one or both eligibility gates. Not scored. Specific rejection reason provided.

Batch Processing

Large uploads (20+ resumes) are processed in batches of 5 to stay within the AI model's context limits. Each batch is scored independently, then all results are merged and re-ranked together. This makes the system reliable at any scale — whether you upload 3 resumes or 50.

AI Accuracy Validation

The AI scoring model was validated against manual screening results performed independently. In a test with 17 resumes for a QC Engineer role:

  • Manual review identified 5 qualified candidates out of 17 — rejecting 12 for not meeting the B.Tech/B.E. education requirement
  • AI scoring (after calibration) identified the same 5 qualified candidates and rejected the same 12, with matching rejection reasons
  • Score differentiation was strong — qualified candidates received varied scores (56 to 89) rather than being clustered at the same number, reflecting genuine differences in fit
Key takeaway: The AI doesn't just save time — it produces the same outcome as a careful human reviewer, but consistently across every opening, every batch, every time. The gates ensure that basic requirements are enforced without exception, while the weighted scoring produces meaningful differentiation among qualified candidates.

Product Iterations & Learnings

Building AI-powered features requires iteration. The scoring model, in particular, went through several rounds of refinement before achieving accuracy on par with manual screening. Below are the key challenges encountered and how each was resolved.

● AI Scored Unqualified Candidates as "Good Match"

▼

Observation: In early testing, Diploma holders were receiving scores of 60–75 for roles requiring B.Tech/B.E. — appearing as "Good match" when they should have been rejected outright. Manual screening rejected these same candidates immediately.

Root cause: The AI was treating the education requirement as a soft preference rather than a hard gate. It gave partial credit for having a related qualification at a lower level.

Resolution: We implemented an explicit degree hierarchy in the AI instructions (10th/12th < ITI < Diploma < B.Tech/B.E. < M.Tech) with concrete rejection examples. A "lower degree never satisfies a higher degree requirement" rule was added with repeated emphasis. Post-fix, the AI's rejections matched manual screening exactly.

Impact: This was the single highest-impact change to scoring accuracy. Without it, the tier system was unreliable — unqualified candidates were consuming Tier 2/3 slots meant for qualified ones.

● Scoring Lacked JD Requirement Prioritisation

▼

Observation: The AI was treating all JD requirements equally. A candidate who matched "nice to have" skills but missed "mandatory" ones could score similarly to one who matched the must-haves. This produced misleading rankings.

Resolution: We introduced a three-tier JD requirement weighting system (Must-have = 3x, Should-have = 2x, Nice-to-have = 1x) based on signal words in the JD text. The AI now parses and classifies every requirement before scoring, ensuring that must-have gaps cost significantly more than nice-to-have gaps.

Impact: Score differentiation improved. Candidates with strong must-have alignment now clearly outscore those with only peripheral matches.

● PDF Resumes Returned Garbled Text

▼

Observation: In the initial architecture, PDF files were being read in the browser before sending to the server. Browsers returned raw binary data instead of readable text, making every resume unprocessable.

Resolution: Moved PDF parsing entirely to the server using a dedicated library. The browser now uploads the raw file, and the server extracts clean text. This also resolved a secondary issue where the v2 PDF library had a different API than expected (locked to v1.1.1).

Impact: Went from 0% successful resume reads to 100%. This was a blocker — without working PDF extraction, the entire AI scoring feature was non-functional.

● Conflicting AI Instructions Produced Confused Results

▼

Observation: The AI was receiving two separate sets of scoring instructions — one from an external file and one built into the server code. The two sets had conflicting criteria, causing the AI to produce inconsistent, unreliable scores.

Resolution: Removed the external instruction file entirely. Consolidated everything into a single, self-contained prompt embedded in the server code. This eliminated ambiguity and ensured the AI always operates from one authoritative set of rules.

Impact: Scoring consistency improved immediately. The AI now applies the same logic to every batch without variation.

● Large Batches Exceeded AI Context Limits

▼

Observation: Uploading 20+ resumes at once caused failures because the combined text exceeded the AI model's input capacity.

Resolution: Implemented batch processing — resumes are grouped into batches of 5, scored independently, then merged and re-ranked. This scales linearly regardless of upload size.

Impact: The system now handles any number of resumes reliably. Upload 50 at once and get ranked results for all of them.

● Rejected Candidates Were Mixed Into Tier 3

▼

Observation: Candidates with a score of 0 (rejected for not meeting basic requirements) were being grouped with qualified candidates and placed into Tier 3. This diluted the tier percentages and mixed unqualified candidates with low-scoring but eligible ones.

Resolution: Added a separate "Rejected" category. The system now removes rejected candidates before tier assignment, then distributes only qualified candidates across Tiers 1, 2, and 3.

Impact: Tier assignments became meaningful. HR can trust that every candidate in any tier has passed the basic eligibility checks.

● JD Database Had Incorrect Placeholder Data

▼

Observation: The portal auto-populated fields from a database that contained sample/placeholder data, not actual JD content. "Billing Engineer" showed Finance department and Mumbai location when the real JD says Civil/Billing and Gurugram.

Resolution: Extracted data from the official Word documents (Saavira_JDs_All_Positions.docx and Mohali Hiring Questionnaire) and replaced all 20 affected entries with verified, field-accurate data.

Impact: Every auto-populated field now matches the official company documents. This is foundational — if the JD data is wrong, everything downstream (screening, scoring, outreach) is built on incorrect information.

● Database Library Required Windows-Specific Build Tools

▼

Observation: The initially chosen database library (better-sqlite3) required Visual Studio C++ Build Tools to install on Windows, creating an unnecessary dependency barrier.

Resolution: Switched to sql.js — a pure JavaScript SQLite implementation that requires no native compilation. Same functionality, zero installation friction.

Impact: The server can be set up with a single npm install on any machine. No special tools required.

Conclusion

The Job Portal transforms recruitment from a manual, fragmented process into a structured, AI-assisted pipeline. The core value proposition is simple: the same screening accuracy as a careful human reviewer, delivered in minutes instead of hours, with perfect consistency across every opening.

Phase 1 solved the workflow problem — how to create and distribute job openings efficiently. Phase 2 solved the evaluation problem — how to screen candidates accurately at scale. Together, they form a complete hiring operations platform that takes a role from "we need to hire someone" to "here are the top candidates to call" with minimal manual effort.

The AI scoring model has been calibrated through multiple iterations and validated against manual screening results. The eligibility gate system ensures that basic requirements are enforced without exception, while the weighted scoring model (with JD requirement prioritisation) produces meaningful differentiation among qualified candidates. The result is a system that HR can trust to produce reliable, defensible shortlists.

With 41 pre-loaded roles, 7 distribution channels, a three-layer AI scoring model, batch processing for scale, persistent storage, and a complete candidate pipeline from upload to outreach — the portal is ready for production deployment.
